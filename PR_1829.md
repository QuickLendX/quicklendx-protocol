Lock in the pagination boundary `offset=0, offset==count, limit==MAX_QUERY_LIMIT` with explicit unit tests on the `pagination` module. ### Key Changes

1. **Wired `pagination.rs` into the crate** (`pub mod pagination;` at `lib.rs:89`) тАФ the module was already present on disk but never declared, so its helpers were unreachable from tests.

2. **Wired `test_queries.rs` as a test module** (`#[cfg(test)] mod test_queries;` at `lib.rs:164`) тАФ the existing pagination unit tests and escrow query consistency tests are now compiled and run under `cargo test`.

3. **Removed `extern crate alloc;` from `pagination.rs`** тАФ this was the root cause of the CI failure. The crate root (`lib.rs:33`) already declares `extern crate alloc;`, and Rust 2021 edition permits `use alloc::...` in submodules without a redundant `extern crate` declaration. The duplicate declaration interfered with Soroban's `#[contractimpl]` macro allocator setup, emitting 19 cascading compilation errors.

4. **Added 4 targeted tests** (`test_queries.rs` section 10) covering the boundary where all three values coincide at `MAX_QUERY_LIMIT`:
   - `test_offset_zero_limit_max_on_full_collection_has_no_more` тАФ `validate_pagination_params(0, 100, 100)` reports `has_more=false`
   - `test_offset_zero_limit_max_on_full_collection_returns_all` тАФ `paginate_slice` returns all 100 items; `calculate_safe_bounds` тЖТ `(0, 100)`
   - `test_offset_equals_count_limit_max_returns_empty` тАФ `offset=100, limit=100, total=100` yields empty result, `has_more=false`
   - `test_cross_consistency_at_max_query_limit_boundary` тАФ both helpers agree at `(0, 100, 100)` and `(100, 100, 100)`
