## Bid Data Structure

Defined in `src/types.rs:176`:

```rust
pub struct Bid {
    pub bid_id: BytesN<32>,          // Unique: 0xB1 0xD0 prefix + timestamp + counter + mix fill
    pub invoice_id: BytesN<32>,      // Linked invoice
    pub investor: Address,           // Bidder address (requires auth)
    pub bid_amount: i128,            // Capital offered (must be > 0)
    pub expected_return: i128,       // Investor-specified return expectation
    pub timestamp: u64,              // Ledger timestamp at placement
    pub status: BidStatus,           // Current lifecycle state (see §2)
    pub expiration_timestamp: u64,   // Auto-computed: `now + TTL_days * 86400`
}
```

The `bid_id` is generated by `BidStorage::generate_unique_bid_id()` (`bid.rs:1049`).
It is allocated at placement time and never changes for the bid's lifetime.

## BidStatus State Machine

Defined in `src/types.rs:30`:

```rust
pub enum BidStatus {
    Placed,     // Active — can be accepted, withdrawn, cancelled, or expire
    Accepted,   // Terminal — bid was accepted by the business (investment created)
    Withdrawn,  // Terminal — investor withdrew before acceptance
    Expired,    // Terminal — TTL elapsed without acceptance (auto-transition)
    Cancelled,  // Terminal — investor cancelled before acceptance
}
```

### Transition Diagram (ASCII)

```
                    ┌──────────────────────────────────────────┐
                    │               Placed                     │
                    │    (active, only mutable state)          │
                    └────┬──────┬──────────┬──────────┬───────┘
                         │      │          │          │
              ┌──────────┘      │          │          └──────────────┐
              ▼                 ▼          ▼                         ▼
         ┌────────┐      ┌─────────┐  ┌────────┐            ┌──────────┐
         │Accepted│      │Withdrawn│  │Expired │            │Cancelled │
         │terminal│      │terminal │  │terminal│            │terminal  │
         └────────┘      └─────────┘  └────────┘            └──────────┘
```

### Valid vs Invalid Transitions

| From       | To          | Valid? | Trigger                                        |
|------------|-------------|--------|-------------------------------------------------|
| `Placed`   | `Accepted`  | Yes    | `accept_bid` (lib.rs:1407)                      |
| `Placed`   | `Withdrawn` | Yes    | `withdraw_bid` (lib.rs:1285)                    |
| `Placed`   | `Cancelled` | Yes    | `cancel_bid` (lib.rs:1273)                      |
| `Placed`   | `Expired`   | Yes    | Auto during `cleanup_expired_bids` or `refresh_*` |
| `Accepted` | *any*       | No     | Terminal — immutable                            |
| `Withdrawn`| *any*       | No     | Terminal — immutable                            |
| `Expired`  | *any*       | No     | Terminal — immutable                            |
| `Cancelled`| *any*       | No     | Terminal — immutable                            |

**Race safety**: `cancel_bid` (bid.rs:979) and `withdraw_bid` (lib.rs:1285)
both re-read the bid after `require_auth` to guard against concurrent transitions.
If the bid is no longer `Placed`, they return `false` / `OperationNotAllowed`.

### Expiry Determination

```rust
// bid.rs:113–115
pub fn is_expired(&self, current_timestamp: u64) -> bool {
    current_timestamp > self.expiration_timestamp
}
```

A bid is **still valid at** `expiration_timestamp` exactly, and becomes expired
starting from the next second. All cleanup and acceptance paths use this same
strict `>` predicate, avoiding off-by-one divergence across call sites.

## Full Lifecycle: Place Through Terminal

### 1. Place (`lib.rs:1319` — `place_bid`)

Pre-conditions enforced in order:

1. Protocol is not paused (`pause::PauseControl::require_not_paused`)
2. Investor authorizes (`investor.require_auth()`)
3. `bid_amount > 0` (`InvalidAmount` if not)
4. Invoice exists and is `Verified` (`InvoiceNotFound` / `InvalidStatus`)
5. Invoice currency is whitelisted (`require_allowed_currency`)
6. Investor is verified and not exceeding their investment limit
7. Expired bids are cleaned first (`cleanup_expired_bids`)
8. `MAX_BIDS_PER_INVOICE` (50) not exceeded (`MaxBidsPerInvoiceExceeded`)
9. Per-investor active-bid limit not exceeded (`OperationNotAllowed`)
10. `validate_bid(...)` passes (amount, expected return, ownership, risk checks)

On success:

- `bid_id` is generated (`generate_unique_bid_id`)
- `expiration_timestamp = now + (TTL_days * 86400)`
- Status = `Placed`
- Stored + indexed (invoice index, investor index, global index)
- Event `emit_bid_placed` fires

### 2. Accept (`lib.rs:1407` — `accept_bid`)

Pre-conditions:

1. Protocol not paused
2. Business authorizes (`invoice.business.require_auth()`)
3. Business KYC is not pending
4. Invoice is `Verified`, bid is `Placed`
5. Cleanup runs before status check

On success (`accept_bid_impl`, lib.rs:1418):

- `Escrow` created (investor → escrow contract)
- Bid transitions `Placed → Accepted`
- Invoice transitions `Verified → Funded`
- `Investment` record created (`Active`)
- Events: `emit_escrow_created`, `emit_bid_accepted`

### 3. Withdraw (`lib.rs:1285` — `withdraw_bid`)

- Investor authorizes, KYC not pending
- Atomically validates `status == Placed` after auth
- Transitions `Placed → Withdrawn`
- Emits `emit_bid_withdrawn`

### 4. Cancel (`lib.rs:1273` / `bid.rs:979` — `cancel_bid`)

- Investor authorizes
- Atomically validates `status == Placed`
- Transitions `Placed → Cancelled`
- Returns `false` if bid not found or already not `Placed`

### 5. Expire (Automatic)

No direct entrypoint. Expiry is a **passive transition**:

- `cleanup_expired_bids` / `refresh_expired_bids` scans bids, transitions any
  `Placed` bid with `current_timestamp > expiration_timestamp` to `Expired`
- `is_expired()` is also evaluated directly by `accept_bid` and by `into_flight`
  for expired bid guards
- Every bid storage path that evaluates `Placed` status checks expiry first

## Deterministic Bid Ranking

Defined in `bid.rs:857` — `compare_bids`. Sorting is **strictly deterministic**
so every validator computes the same acceptance order.

### Five-Level Comparator Chain

Higher values win at each level. The next level is only evaluated when the
current level is a tie.

| Level | Field              | Computation                        | Direction                   |
|-------|--------------------|------------------------------------|-----------------------------|
| 1     | **Profit**         | `expected_return − bid_amount`     | Higher profit wins          |
| 2     | **Expected Return**| `expected_return`                  | Higher return wins          |
| 3     | **Bid Amount**     | `bid_amount`                       | Higher amount wins          |
| 4     | **Timestamp**      | `timestamp`                        | Newer (higher) wins         |
| 5     | **Bid ID**         | Byte-array lexicographic comparison| Byte-wise comparison wins   |

```rust
// bid.rs:857–877
pub fn compare_bids(bid1: &Bid, bid2: &Bid) -> Ordering {
    let profit1 = bid1.expected_return.saturating_sub(bid1.bid_amount);
    let profit2 = bid2.expected_return.saturating_sub(bid2.bid_amount);
    if profit1 != profit2 { return profit1.cmp(&profit2); }
    if bid1.expected_return != bid2.expected_return { return bid1.expected_return.cmp(&bid2.expected_return); }
    if bid1.bid_amount != bid2.bid_amount { return bid1.bid_amount.cmp(&bid2.bid_amount); }
    if bid1.timestamp != bid2.timestamp { return bid1.timestamp.cmp(&bid2.timestamp); }
    if bid1.bid_id != bid2.bid_id { return bid1.bid_id.to_array().cmp(&bid2.bid_id.to_array()); }
    Ordering::Equal
}
```

### Design Rationale for Each Level

1. **Profit**: The protocol's economic goal — maximize the business's effective discount
   (profit = what the investor expects back minus what they put in).
2. **Expected Return**: If two bids yield identical profit, prefer the one with
   the higher face value (more capital circulating).
3. **Bid Amount**: Larger bids fund more of the invoice.
4. **Timestamp**: If all economic values match, the newer bid wins (the fresher
   investor signal is preferred).
5. **Bid ID**: Final stable tiebreaker. `bid_id` embeds `(timestamp, counter)`
   and is guaranteed unique. This prevents validator-dependent ordering from
   storage iteration or hash-dependent iteration.

### Key Functions

| Function                                   | Location     | Description                                    |
|--------------------------------------------|--------------|------------------------------------------------|
| `compare_bids(b1, b2) → Ordering`          | bid.rs:857   | Pure comparator implementing the five-level chain |
| `rank_bids(env, invoice_id) → Vec<Bid>`    | bid.rs:944   | All placed bids sorted best→worst              |
| `get_best_bid(env, invoice_id) → Option<Bid>`| bid.rs:934 | Top-ranked placed bid (≡ `rank_bids[0]`)       |

**Invariant**: `get_best_bid(...)` always returns the same bid as `rank_bids(...).get(0)`.

### Exclusion Rules

Only bids with `status == BidStatus::Placed` participate in ranking.
`Accepted`, `Withdrawn`, `Expired`, and `Cancelled` bids are excluded.

`rank_bids` calls `get_bid_records_for_invoice`, which calls `refresh_expired_bids`
first — so expired `Placed` bids are transitioned to `Expired` before ranking.

## Worked Ranking Example

Consider an invoice with four competing `Placed` bids. All values in smallest
currency units (e.g., stroops).

### Bid Profiles

| Bid   | bid_amount | expected_return | Profit (L1) | timestamp (L4) |
|-------|------------|-----------------|-------------|----------------|
| **A** | 1000       | 1100            | **100**     | 1700000000     |
| **B** | 800        | 920             | **120**     | 1700000100     |
| **C** | 1000       | 1100            | **100**     | 1700000000     |
| **D** | 1000       | 1100            | **100**     | 1700000050     |

### Step-by-Step Resolution

**Level 1 — Profit**:

| Bid | Profit |
|-----|--------|
| A   | 100    |
| B   | **120** ← highest |
| C   | 100    |
| D   | 100    |

B is the leader. A, C, D are tied at profit 100.

**Level 1 resolves B > A, B > C, B > D** — B is ranked #1 overall.

**Level 1 tie between A, C, D** → proceed to Level 2.

---

**Level 2 — Expected Return** (among A, C, D):

| Bid | expected_return |
|-----|-----------------|
| A   | 1100            |
| C   | 1100            |
| D   | 1100            |

All three are tied → proceed to Level 3.

**Level 3 — Bid Amount** (among A, C, D):

| Bid | bid_amount |
|-----|------------|
| A   | 1000       |
| C   | 1000       |
| D   | 1000       |

Still tied → proceed to Level 4.

**Level 4 — Timestamp** (among A, C, D):

| Bid | timestamp   |
|-----|-------------|
| A   | 1700000000  |
| C   | 1700000000  |
| D   | **1700000050** ← newest |

D is newer than A and C → **D ranks #2**.

A and C are still tied on all economic values and timestamp → proceed to Level 5.

**Level 5 — Bid ID lexicographic comparison** (A vs C):

Since A and C have identical profit (100), expected_return (1100), bid_amount
(1000), and timestamp (1700000000), the final tiebreaker is their `bid_id` bytes:

| Bid | bid_id (hex prefix)                                              |
|-----|------------------------------------------------------------------|
| A   | `B1 D0 00 00 00 00 65 53 8D 80 00 00 00 00 00 00 00 01 ...`    |
| C   | `B1 D0 00 00 00 00 65 53 8D 80 00 00 00 00 00 00 00 03 ...`    |

C's counter byte at position 17 is `0x03` vs A's `0x01` → C's bid_id is
lexicographically greater → **C ranks #3**, A ranks #4.

### Final Ranking

```
1. B  (profit 120)
2. D  (profit 100, timestamp newest among tied)  
3. C  (profit 100, bid_id tiebreaker over A)
4. A  (profit 100, lowest bid_id)
```

### Key Takeaway

The bid_id tiebreaker ensures a **fully deterministic total order** even when
two different investors submit bids with identical economics at the exact same
ledger timestamp. Without Level 5, the iteration order of the storage Vec could
produce different rankings on different validator nodes.

## TTL Configuration

### Constants

Defined in `bid.rs:36–38`:

| Constant                | Value | Meaning                                   |
|-------------------------|-------|-------------------------------------------|
| `DEFAULT_BID_TTL_DAYS`  | 7     | Default TTL when no admin override is set |
| `MIN_BID_TTL_DAYS`      | 1     | Minimum allowed TTL (prevents immediate expiry) |
| `MAX_BID_TTL_DAYS`      | 30    | Maximum allowed TTL (prevents indefinitely locked funds) |
| `SECONDS_PER_DAY`       | 86400 | Conversion factor (not configurable)       |

### TTL Enforcement

A value of `0` is explicitly rejected with `InvalidBidTtl` (bid.rs:312–313).
Values outside `[1, 30]` are also rejected (bid.rs:315–316).

### Admin Entrypoints

| Entrypoint                       | Location     | Description                                           |
|----------------------------------|--------------|-------------------------------------------------------|
| `set_bid_ttl_days(days)`         | lib.rs:503   | Set TTL to `days` (must be 1–30, not 0)              |
| `get_bid_ttl_days()`             | lib.rs:510   | Read active TTL (falls back to 7)                     |
| `get_bid_ttl_config()`           | lib.rs:515   | Full snapshot: current, min, max, default, is_custom  |
| `reset_bid_ttl_to_default()`     | lib.rs:520   | Remove override, revert to 7 days                     |

All setters require admin authorization and emit `ttl_upd` events
(`emit_bid_ttl_updated` in bid.rs:321) with old value, new value, admin address,
and ledger timestamp.

### TTL Config Snapshot

`BidTtlConfig` (bid.rs:60–72) provides a complete picture in one call:

### Expiration Calculation

Each bid computes its `expiration_timestamp` at placement using the active TTL:


## Limits & Ceilings

### MAX_BIDS_PER_INVOICE

| Constant               | Value | Defined     |
|------------------------|-------|-------------|
| `MAX_BIDS_PER_INVOICE` | 50    | bid.rs:49   |

Enforced in `place_bid` (lib.rs:1361–1364). When the count of `Placed` bids on
an invoice reaches 50, new placements are rejected with `MaxBidsPerInvoiceExceeded`.

Bids in terminal states (`Expired`, `Cancelled`, `Withdrawn`, `Accepted`) are
**not** counted toward this limit. The cap is against concurrent *active* bids.

### Per-Investor Active-Bid Limit

| Constant                              | Value | Defined     |
|---------------------------------------|-------|-------------|
| `DEFAULT_MAX_ACTIVE_BIDS_PER_INVESTOR`| 20    | bid.rs:41   |
| `INVESTOR_BID_LIMIT_DISABLED`         | 0     | bid.rs:52   |

Enforced in `place_bid` (lib.rs:1366–1371). Counts only `Placed` (non-expired)
bids across **all** invoices for a given investor. If the limit is set to `0`,
enforcement is disabled entirely — any number of open bids is allowed.

| Entrypoint                              | Location     | Description                                      |
|-----------------------------------------|--------------|--------------------------------------------------|
| `get_max_active_bids_per_investor()`    | lib.rs:526   | Read active limit (defaults to 20)               |
| `set_max_active_bids_per_investor(limit)`| lib.rs:531  | Admin sets limit; 0 disables enforcement         |
| `count_active_placed_bids_for_investor()` | bid.rs:516  | Counts current `Placed` bids (prunes expired first) |

`BidLimitConfig` (bid.rs:89–98) provides a full snapshot analogous to `BidTtlConfig`:

```rust
pub struct BidLimitConfig {
    pub limit: u32,        // Active cap (0 = disabled)
    pub default_limit: u32,// 20
    pub is_disabled: bool, // true when limit == 0
    pub is_custom: bool,   // true when admin has overridden
}
```

### Expired Bids and Limit Counting

The per-investor limit counts only `Placed` bids whose `expiration_timestamp`
has **not** passed. `count_active_placed_bids_for_investor` (bid.rs:516) calls
`refresh_investor_bids` first, which transitions expired `Placed` bids to
`Expired` and prunes them from the investor's active index. This keeps the
count accurate and prevents stale expired bids from blocking new placements.

## Cleanup: `cleanup_expired_bids` vs `cleanup_expired_bids_paged`

Both functions scan an invoice's bid list, transition expired `Placed` bids to
`Expired`, and compact the index to remove entries in terminal `Expired` state
(and orphaned entries where the bid no longer exists in storage).

### `cleanup_expired_bids` (bid.rs:670)

```rust
pub fn cleanup_expired_bids(env: &Env, invoice_id: &BytesN<32>) -> u32
```

- **Batch size**: All bids (up to `MAX_BIDS_PER_INVOICE` = 50)
- **Returns**: Count of bids cleaned
- **Idempotent**: Second call at the same ledger time returns 0
- **Use when**: Bid count is well under 50, or called inline during `place_bid` / `accept_bid`

### `cleanup_expired_bids_paged` (bid.rs:716)

```rust
pub fn cleanup_expired_bids_paged(
    env: &Env,
    invoice_id: &BytesN<32>,
    offset: u32,
    limit: u32,
) -> (u32, u32)  // (cleaned_count, total_remaining)
```

- **Batch size**: Up to `limit` bids (capped at `MAX_BIDS_PER_INVOICE`)
- **Returns**: `(cleaned_count, total_remaining)` tuple
- **Use when**: An invoice is at or near the 50-bid ceiling and a single-call
  cleanup could approach Soroban instruction budget limits

### Instruction Budget Guidance

| `limit` param | Approx. instructions | Safety level        |
|---------------|----------------------|---------------------|
| 10            | ~100–200             | Very safe           |
| 25            | ~250–500             | Safe                |
| 50            | ~500–1000            | Worst-case, may approach budget |

### When to Use Each

| Scenario                                          | Recommended function           |
|---------------------------------------------------|--------------------------------|
| `place_bid` / `accept_bid` inline cleanup         | `cleanup_expired_bids` (automatic) |
| Off-chain indexer proactively maintaining state   | `cleanup_expired_bids`         |
| Operator processing an invoice at/near 50-bid cap | `cleanup_expired_bids_paged`   |
| Cron job splitting work across multiple txns      | `cleanup_expired_bids_paged`   |

Both functions are **fully idempotent** and **never touch terminal bids**
(`Accepted`, `Withdrawn`, `Cancelled`). Only `Placed` bids past their
`expiration_timestamp` are transitioned to `Expired`; already-`Expired` bids
are removed from the invoice index but their bid record in storage is preserved.

### Cleanup Invariants (bid.rs:544–555)

1. Terminal bids (`Accepted`, `Withdrawn`, `Cancelled`) are **never** modified or removed
2. Active `Placed` bids are preserved if not yet expired
3. Expired/orphaned bids are removed from the index to prevent unbounded growth
4. The operation is idempotent — calling multiple times on same state yields same result
5. Cleanup is bounded by O(N) compute and storage changes (N ≤ `MAX_BIDS_PER_INVOICE`)
